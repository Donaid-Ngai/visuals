# Context and intent

## Goal
Make the operational improvement immediately legible to a non-technical audience.

The page should answer two questions quickly:
1. Why is this better than the old way?
2. What actually changed underneath?

## Required communication order
1. Show the pain first
2. Show the improved workflow second
3. Reveal the connected system structure underneath
4. Make ongoing operating costs explicit

## What must remain true
- The old process should feel visibly longer and more annoying.
- The new process should feel compact, structured, and accountable.
- The system diagram should preserve visible subsystem connections.
- Logging and board-wide visibility should be explicit, not implied.
- Cost framing should read as capability upkeep, not random software spend.

## Audience assumptions
This page is for a business / board-facing audience, not an engineering audience.

It should remain concrete and operational, while still showing enough structure to explain where the improvement comes from.
